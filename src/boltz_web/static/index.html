关系说明

一个用户可以有多个项目；每个项目下面有自己的资产和任务；资产代表可复用的数据对象，文件是资产下面保存的具体 PDB、SDF、CSV 或结果文件。

  • 项目：一次研究课题或一个工作空间，例如某个靶点、某个系列化合物。
  • 资产：项目内可复用的蛋白、配体、复合物、结果等数据对象。
  • 任务：对资产执行的计算流程，例如蛋白准备、配体准备、对接、FEP。
  • 文件：资产或任务实际产生/引用的磁盘文件，可作为后续任务输入。

TPD / PROTAC

面向 targeted protein degradation 的三元复合物与 linker 设计入口。它和普通 docking 不同，需要 POI、E3 ligase、warhead、E3 ligand 和 linker 的专门对象模型。

任务 / Task构建 POI-E3 三元复合物并设计 PROTAC linker。Model ternary complexes and design linkers.
药设作用 / Role支持 targeted protein degradation 项目的构象和 linker 决策。Support TPD ternary-complex and linker decisions.
输入 / InputsPOI 口袋、E3 ligase、warhead、E3 ligand。POI pocket, E3 ligase, warhead, and E3 ligand.
输出 / OutputsPROTAC 候选、linker 采样、三元复合物模型。PROTAC candidates, linker samples, and ternary models.

TPD 输入

TPD 需要的能力

  • POI/E3 二元结构准备与口袋识别。
  • linker 长度、柔性、连接位点和构象采样。
  • 三元复合物构建、碰撞过滤、界面稳定性评分。
  • 输出 PROTAC 候选分子和三元复合物模型。

TPD 建议作为独立 workflow，不和常规 docking 混在一个页面里。

SAR / 构效关系

面向药化决策的化合物系列分析入口，用实验活性、配体结构、对接结果和 FEP 结果建立可解释的趋势视图。

任务 / Task整合活性、结构和计算结果做构效分析。Analyze SAR from activity, structure, and computed results.
药设作用 / Role指导下一轮设计、取代基优先级和系列优化。Guide next-round design and series optimization.
输入 / InputsSMILES/结构表、活性数据、对接/FEP 结果。Chemical tables, activities, docking/FEP results.
输出 / OutputsSAR 趋势、MMPA、R-group、设计建议。SAR trends, MMPA, R-groups, and design ideas.

数据导入

后续会把表格转为 SAR 数据资产，并与项目内配体、对接和 FEP 结果合并。

输入和输出关系

  • 输入：配体资产、活性表、对接打分、FEP 结果、ADMET 字段。
  • 处理：标准化字段、按系列分组、计算相似性/MCS/MMPA。
  • 输出：SAR 分析资产、趋势图、候选设计列表和报告文件。

主工作区 / SAR 分析视图

右侧用于最大化分子表格、骨架/R-group、活性趋势、MMPA 和下一轮设计视图。

  • 分子表格：SMILES、活性、选择性、ADMET 和计算结果字段。
  • Scaffold / R-group 分组：按骨架和取代基观察趋势。
  • 活性分布：pIC50、Ki、Kd、EC50 等字段的排序和分布图。
  • Matched Molecular Pair：定位小结构变化带来的活性变化。
  • 下一轮设计：基于 SAR 趋势生成待合成/待计算候选。

自动化链路 / Automation Flow

  1. 创建/选择项目：`POST /api/v1/projects` → `project_id`。
  2. 导入蛋白：`POST /api/v1/assets/proteins/pdb` 或 `POST /api/v1/assets/upload` → protein `asset_id`。
  3. 创建口袋：`POST /api/v1/assets/pockets` → pocket `asset_id`。
  4. 创建/上传/编辑配体：`/assets/ligands/*` → ligand `asset_id`。
  5. 准备配体：`POST /api/v1/preparations/ligand` → prepared_ligand `asset_id`。
  6. 生成 Boltz input：`POST /api/v1/boltz-inputs` → boltz_prediction_input `asset_id`。
  7. 创建/查询任务：`POST /api/v1/jobs`、`GET /api/v1/jobs`、`GET /api/v1/jobs/{job_id}/events`。
  8. 下载文件：`GET /api/v1/assets/{asset_id}/files/{file_id}/download`。

完整接口、请求体和响应结构从 FastAPI OpenAPI 自动读取；下面的手写示例只解释自动化工作流如何衔接。
