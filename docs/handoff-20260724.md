## Persistent directories

The web stack must use a host-mounted project directory:

```bash
BOLTZ_DATA_HOST_DIR=/data/ssd/jhu/boltz-web/data
```

The compose file mounts it to `/data` and sets:

```bash
BOLTZ_CACHE=/data/model-cache
```

This means user projects, uploaded files, generated assets, task outputs, and
Boltz model/cache files are expected to survive container rebuilds and restarts.

## Model source

Do not rely on implicit downloads into a container-local home directory. Download
the mirrored Boltz-2 model files from ModelScope into the persistent cache:

```bash
pip install modelscope
modelscope download \
  --model huluxiaohuowa/boltz-2-mirror \
  --local_dir /data/ssd/jhu/boltz-web/data/model-cache
```

Expected files:

- `boltz2_conf.ckpt`
- `boltz2_aff.ckpt`
- `mols.tar`

local checkout or the shallow `/Users/hulu` search performed during handoff, so
no local model file was removed.

## Known gaps / next work

- The web ligand editor now uses bundled Ketcher assets in the `boltz-web`
  image. Verify the deployed page after each rebuild because Ketcher is a large
  frontend bundle.
- Protein-preparation chemistry execution is not fully wired into a queue yet.
  Current web protein preparation is text-level PDB cleanup and asset creation;
  the protein-prep worker image provides the chemistry runtime and health loop,
  but queue consumption is reserved for the next integration slice.
- The current protein-prep conda environment pulled OpenMM runtime dependencies
  including CUDA runtime packages, so the ARM image is large. This does not make
  it a PyTorch/GPU worker; it should be optimized later by pinning a stricter
  CPU-only OpenMM/PDBFixer dependency set or splitting light/full prep images.
- A separate ligand-prep worker image has not been implemented yet. Current
  ligand preparation runs in the web service through RDKit/Ketcher endpoints.
- The current Boltz input generator produces `input.yaml`; it does not run the
  Boltz model job yet.
- The mistaken SWR repository name from the first push attempt may still exist
  in the registry backend. It is not referenced by compose or README. Remove it
  from SWR console/API if repository cleanup is required.
